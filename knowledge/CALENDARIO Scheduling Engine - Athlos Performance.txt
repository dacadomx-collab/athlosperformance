Con el contexto de Athlos y las recomendaciones previas sobre el flujo operativo del laboratorio, considero que el calendario no debe diseñarse como un simple "agenda de citas", sino como un Scheduling Engine desacoplado del negocio. Es decir, un motor genérico de disponibilidad, reservas, sincronización y reglas que posteriormente pueda reutilizarse en cualquier sistema (clínicas, consultorios, laboratorios, spas, coworkings, despachos, etc.). Esto mantiene la filosofía de convertir el calendario en un componente de infraestructura y no en un módulo específico de un proyecto. Esa aproximación además es consistente con la arquitectura modular que ya hemos venido definiendo para el sistema.
________________


1. Arquitectura recomendada (Event Driven + Sync Engine)
No recomiendo que el calendario escriba directamente sobre Google Calendar o Apple Calendar.
Recomiendo introducir una capa intermedia.
                  Frontend


         FullCalendar / Mobile Calendar


                     │


             Calendar API Gateway


                     │


        ┌────────────┴─────────────┐


 Reservation Engine         Availability Engine


        │                           │


        └────────────┬──────────────┘


             Calendar Event Bus


                     │


      ┌──────────────┼──────────────┐


 Google Sync      Apple Sync     Notification Engine


      │                │                │


Google Calendar     ICS Feed       WhatsApp
OAuth               CalDAV         Email
Webhooks            webcal://      SMS


Esta separación ofrece varias ventajas:
* desacopla completamente el proveedor de calendario
* facilita agregar Microsoft Outlook en el futuro
* evita dependencias directas del frontend
* centraliza las reglas de negocio
* simplifica auditoría

14. Especificación del "Módulo de Calendario Genérico" (Plantilla Maestra)
Para convertir este componente en un activo reutilizable dentro de cualquier ecosistema, recomiendo documentarlo como un módulo independiente con las siguientes secciones:
Sección
	Contenido
	Objetivo
	Gestión de disponibilidad, reservas y sincronización omnicanal.
	Actores
	Administrador, operador, especialista, cliente y servicios externos.
	Entidades
	Calendario, recurso, franja horaria, reserva, cita, participante, sincronización, notificación.
	Reglas de negocio
	Horarios, capacidad, buffers, cancelaciones, recurrencias, permisos y prioridades.
	Casos de uso
	Crear, mover, cancelar, reprogramar, bloquear horarios, asignar recursos y consultar disponibilidad.
	Motor de disponibilidad
	Algoritmos de generación de slots, validaciones y cálculo de capacidad.
	Motor de sincronización
	Adaptadores para Google, Apple (ICS/CalDAV) y futuros proveedores como Outlook.
	API pública
	Endpoints REST/GraphQL para disponibilidad, reservas, sincronización y webhooks.
	Eventos del dominio
	AppointmentCreated, AppointmentUpdated, AppointmentCancelled, AvailabilityChanged, SyncCompleted, ConflictDetected.
	UX/UI
	Patrones para escritorio (matriz semanal/mensual), móvil (agenda vertical), accesibilidad, drag & drop y semaforización.
	Seguridad
	OAuth 2.0, RBAC, auditoría, versionado de eventos y trazabilidad completa.
	Extensibilidad
	Adaptadores para nuevos calendarios, motores de notificaciones y reglas de negocio específicas sin modificar el núcleo.
	Recomendación estratégica
Si el objetivo es construir una plataforma con filosofía de "andamiaje empresarial" reutilizable, diseñaría este componente como un Calendar Scheduling Engine (CSE), completamente desacoplado del dominio. El laboratorio, una clínica, un despacho legal o una empresa de servicios consumirían exactamente el mismo motor mediante configuración. La lógica de horarios, capacidad, sincronización, recurrencias, conflictos y disponibilidad viviría en el CSE; las reglas específicas de cada negocio (por ejemplo, "máximo 4 personas por hora" o "cancelación válida con 3 horas de anticipación") serían políticas configurables. Así el calendario deja de ser un módulo de Athlos y se convierte en una pieza de infraestructura reutilizable para cualquier sistema futuro.
